### 13. array 和 slice 的区别

> [!TIP]
> **核心回答：**
> 在 Go 语言中，数组（Array）和切片（Slice）看起来非常相似，但它们的底层设计、内存模型以及使用场景有着天壤之别。
> - **数组**：是固定长度的“铁板一块”，属于值类型。
> - **切片**：是动态可变的“高弹性指针包裹”，属于引用类型。

#### 一、 核心区别速览表

| 维度 | 数组 (Array) | 切片 (Slice) |
| :--- | :--- | :--- |
| **定义方式** | 必须指定长度：`[5]int` 或 `[...]int` | 不能指定长度：`[]int` |
| **长度是否可变** | 固定不变，编译时就已确定 | 动态可变，运行时可自动扩容 |
| **类型是否相同** | `[3]int` 和 `[5]int` 是完全不同的类型 | 只要元素类型相同就是同类型，如 `[]int` |
| **内存结构** | 直接存储所有的值（值类型） | 存储指针、长度、容量（引用类型） |
| **函数传参表现** | 值拷贝（复制整个数组，开销大） | 引用拷贝（仅复制 24 字节的头部结构） |

#### 二、 底层内存模型对比

理解这两者区别的最佳方式是看它们在内存中是如何存储的。
1. **数组**：一块纯粹、连续的数值内存。当你声明 `arr := [3]int{10, 20, 30}` 时，变量 `arr` 紧紧包裹着这 3 个数字。数组变量的地址，就是它第一个元素的地址。
2. **切片**：一个轻量级的结构体（`SliceHeader`）。当你声明 `s := []int{10, 20, 30}` 时，变量 `s` 本身其实是一个占用 24 字节的结构体，它自己不存数据，而是指向一个底层的隐式数组。

切片的 24 字节包含三个字段：
- **Pointer**（指针，8 字节）：指向底层数组中切片开始的那个元素的内存地址。
- **Len**（长度，8 字节）：切片中当前可见的元素个数。
- **Cap**（容量，8 字节）：从切片的起始位置开始，到底层数组末尾的元素个数。

#### 三、 函数传参时的经典面试坑

因为底层结构不同，把数组和切片作为参数传给函数时，Go 语言的处理方式完全不同。Go 语言只有值传递（值拷贝）：

1. **数组传参：复制整栋楼**
   ```go
   func modifyArray(a [5]int) {
       a[0] = 999 // ❌ 修改的是副本，外部数组毫无影响
   }
   ```
   - **问题**：如果数组有 100 万个元素，传参时 Go 会在内存里完整复制 100 万个元素，性能极差。
   - **类型严格**：如果函数定义的是 `[5]int`，你传一个 `[10]int` 进去，编译直接报错。

2. **切片传参：只复制钥匙（指针）**
   ```go
   func modifySlice(s []int) {
       s[0] = 999 // ✅ 修改会直接影响外部，因为共享同一个底层数组
   }
   ```
   - **高效**：无论切片后面挂着 10 个元素还是 1000 万个元素，传参时永远只复制那个 24 字节的 `SliceHeader` 结构体，性能极高且恒定。

#### 四、 代码语法辨析

初学者最容易在语法上搞混它们，这里放一个对比：
```go
// --- 数组的声明方式 ---
a1 := [3]int{1, 2, 3}   // 显式指定长度 3
a2 := [...]int{4, 5, 6} // 编译器根据元素个数自动推导长度为 3

// --- 切片的声明方式 ---
s1 := []int{1, 2, 3}    // 没有长度，这是切片字面量
s2 := make([]int, 3, 5) // 使用 make 创建，长度 3，容量 5
s3 := a1[1:3]           // 从数组截取而来，也是切片
```

> [!NOTE]
> 在实际的开发（尤其是多链、高并发的 Web3 后端）中，99% 的场景我们都在使用切片（Slice），因为它的动态扩容和高效传参特性是业务开发必不可少的。数组通常只在长度绝对固定且不能更改的底层场景使用（例如：以太坊的哈希值 `[32]byte`，或者账户地址 `[20]byte`）。

### 14. JSON 包里使用结构体时，如果不加 tag 能不能正常转成 JSON 字段？

> [!TIP]
> **核心回答：**
> - 如果变量**首字母小写**（private），**无论如何都不能**转成 JSON，因为外部包（包括 `encoding/json`）取不到其反射信息。
> - 如果变量**首字母大写**（public），**不加 tag 可以正常转**为 JSON 字段，此时 JSON 字段名跟结构体内字段原名保持一致。
> - 如果加了 tag，从 struct 转 JSON 的时候，JSON 的字段名就会采用 tag 里的自定义名称。

#### 代码示例

```go
package main

import (
	"encoding/json"
	"fmt"
)

type J struct {
	a string          // 小写无 tag
	b string `json:"B"` // 小写 + tag
	C string          // 大写无 tag
	D string `json:"DD"` // 大写 + tag
}

func main() {
	j := J{
		a: "1",
		b: "2",
		C: "3",
		D: "4",
	}
	fmt.Printf("转为 json 前 j 结构体的内容 = %+v\n", j)
	jsonInfo, _ := json.Marshal(j)
	fmt.Printf("转为 json 后的内容 = %+v\n", string(jsonInfo))
}
```

#### 输出结果

```plaintext
转为 json 前 j 结构体的内容 = {a:1 b:2 C:3 D:4}
转为 json 后的内容 = {"C":"3","DD":"4"}
```

#### 解释说明

结构体里定义了四个字段，分别对应小写无 tag、小写 + tag、大写无 tag、大写 + tag。
转为 JSON 后，首字母小写的字段不管加不加 tag 都不能转为 JSON 里的内容；而首字母大写的字段，加了 tag 可以取别名，不加 tag 则 JSON 内的字段跟结构体字段原名一致。

### 15. 结构体里的字段加了 tag，有什么办法可以获取到这个 tag 的内容？

> [!TIP]
> **核心回答：**
> 结构体字段的 tag 信息可以通过**反射（`reflect` 包）**内的方法来获取。具体步骤是先获取结构体类型的 `reflect.Type`，然后遍历其字段（`Field`），通过 `Field.Tag.Get("tag_name")` 获取对应的 tag 内容。

#### 代码示例

```go
package main

import (
	"fmt"
	"reflect"
)

type J struct {
	a string                             // 小写无 tag
	b string `json:"B"`                  // 小写 + tag
	C string                             // 大写无 tag
	D string `json:"DD" otherTag:"good"` // 大写 + tag
}

func printTag(stru interface{}) {
	// 获取指针指向的值对应的结构体反射类型
	t := reflect.TypeOf(stru).Elem()
	
	// NumField() 可以获得该结构体含有的字段数量
	for i := 0; i < t.NumField(); i++ {
		field := t.Field(i)
		fmt.Printf("结构体内第 %v 个字段 %v 对应的 json tag 是 %v , 还有 otherTag？ = %v \n", 
			i+1, 
			field.Name, 
			field.Tag.Get("json"), 
			field.Tag.Get("otherTag"),
		)
	}
}

func main() {
	j := J{
		a: "1",
		b: "2",
		C: "3",
		D: "4",
	}
	printTag(&j)
}
```

#### 输出结果

```plaintext
结构体内第 1 个字段 a 对应的 json tag 是  , 还有 otherTag？ =  
结构体内第 2 个字段 b 对应的 json tag 是 B , 还有 otherTag？ =  
结构体内第 3 个字段 C 对应的 json tag 是  , 还有 otherTag？ =  
结构体内第 4 个字段 D 对应的 json tag 是 DD , 还有 otherTag？ = good
```

#### 解释说明

1. `printTag` 方法传入的是 `j` 的指针。
2. `reflect.TypeOf(stru).Elem()` 用于获取指针指向的具体结构体类型。
3. `NumField()` 用于遍历结构体内的所有字段。
4. 遍历时，通过 `t.Field(i).Tag.Get("json")` 可以获取到 tag 中为 `json` 的属性值。
5. 如果结构体的字段有多个 tag（如 `otherTag`），同样可以通过相应的 key 获取到对应的属性值。

### 16. slice 深拷贝和浅拷贝的区别？

> [!TIP]
> **核心回答：**
> 理解它们的核心在于：你是仅仅复制了切片的**“外壳”（SliceHeader）**，还是连同它底层的**“肉身”（底层数组）**一起复制了。
> - **浅拷贝**：只复制切片的头部结构（指针、长度、容量），拷贝后的新旧切片共享同一个底层数组。
> - **深拷贝**：在内存中开辟一块完全独立的全新数组空间，并将原数组的值逐个复制过去。

#### 一、 浅拷贝（Shallow Copy）

在 Go 中，有两种非常常见的浅拷贝方式：
- **直接赋值**：`s2 := s1`
- **切片截取**：`s2 := s1[1:3]`

##### 💻 代码示例
```go
package main

import "fmt"

func main() {
	s1 := []int{1, 2, 3}
	// 浅拷贝：s2 只是复制了 s1 的指针、len 和 cap
	s2 := s1
	// 因为共享底层数组，修改 s2 会直接影响 s1
	s2[0] = 99
	fmt.Println("s1:", s1) // 输出: [99, 2, 3]
	fmt.Println("s2:", s2) // 输出: [99, 2, 3]
}
```
- **优点**：极快，性能极高。无论原切片有多大，都只复制 24 个字节。
- **缺点**：两个变量产生强耦合，修改一个会影响另一个，容易在并发或复杂的业务逻辑中引发隐藏 Bug。

#### 二、 深拷贝（Deep Copy）

在 Go 中，标准的深拷贝实现方式是使用内置的 `copy()` 函数。

##### 💻 代码示例
```go
package main

import "fmt"

func main() {
	s1 := []int{1, 2, 3}
	// 1. 必须先为新切片开辟独立的空间，且 len 必须大于等于原切片
	s2 := make([]int, len(s1))
	// 2. 使用内置 copy 函数进行深拷贝
	copy(s2, s1)
	// 3. 修改 s2，s1 完好无损，因为它们在不同的内存地址上
	s2[0] = 99
	fmt.Println("s1:", s1) // 输出: [1, 2, 3] (没变)
	fmt.Println("s2:", s2) // 输出: [99, 2, 3] (变了)
}
```
- **优点**：绝对安全，数据物理隔离。
- **缺点**：伴随着新的内存分配和数据搬运，在切片巨大时会有明显的内存和性能开销。

#### 三、 核心对比表

| 维度 | 浅拷贝 (Shallow Copy) | 深拷贝 (Deep Copy) |
| :--- | :--- | :--- |
| **实现方式** | `s2 := s1` 或 `s1[low:high]` | `make()` 分配空间 + `copy(s2, s1)` |
| **底层数组** | 共享同一个底层数组 | 指向相互独立的两个不同数组 |
| **修改互相影响** | 会（修改新切片会改变老切片） | 不会（完全隔离） |
| **内存开销** | 极小（恒定 24 字节） | 较大（取决于切片的大小） |
| **性能/速度** | 极快（$O(1)$ 时间复杂度） | 较慢（$O(n)$ 时间复杂度，需遍历复制） |

#### ⚠️ 面试大坑：append() 引起的“浅拷贝变深拷贝”

这是一个最经典的切片面试陷阱：**浅拷贝的切片，在执行 append() 之后还会互相影响吗？**
> **答案是**：取决于是否触发了扩容！

```go
package main

import "fmt"

func main() {
	s1 := make([]int, 3, 3) // len=3, cap=3
	s1[0], s1[1], s1[2] = 1, 2, 3

	s2 := s1 // 浅拷贝，此时共享数组

	// 触发 append！由于容量 cap 已经满了(3/3)，s2 内部被迫触发扩容
	s2 = append(s2, 4)

	s2[0] = 99 
	fmt.Println("s1[0]:", s1[0]) // 输出: 1 (s1 未受影响)
}
```
- **原因分析**：在执行 `append` 之前，它们确实是浅拷贝。但当 `append(s2, 4)` 发生时，由于 `s2` 的容量不够了，Go 在底层为 `s2` 重新分配了一块更大的新内存（容量翻倍），并把老数据搬了过去。这就导致 `append` 操作在运行时被动地将原本的“浅拷贝”强制剥离成了“深拷贝”。自此以后，`s1` 和 `s2` 彻底分家。

### 17. map 触发扩容的时机和条件是什么？

> [!TIP]
> **核心回答：**
> Go 的 map 在进行**写操作**（写入新 Key 或修改 Key）时，会检查是否满足以下两个条件之一。只要满足任意一个，就会立刻触发渐进式扩容。

#### 条件一：装载因子超标（触发“翻倍扩容”）

- **触发条件**：map 中的元素总个数 $\text{count}$，除以当前主桶的总个数 $\text{buckets 数量}$，结果大于 6.5。
$$\text{装载因子 (Load Factor)} = \frac{\text{map 中的元素总数}}{\text{当前主桶的总数}} > 6.5$$
- **通俗理解**：每个桶最大容量是 8，如果装载因子超过 6.5，说明每个桶平均已经塞了 6.5 个数据了。
- **底层考量**：如果任由装载因子继续增大，哈希冲突的概率会呈指数级上升，大量的 Key 会被挤到“溢出桶（Overflow Bucket）”里，导致原本 $O(1)$ 的查找性能严重退化成接近 $O(n)$ 的链表遍历。
- **扩容结果**：Go 会直接在内存中开辟一个大小为原来 2 倍的新桶数组（$B$ 变成 $B+1$），并将旧数据分流搬迁过去。

#### 条件二：溢出桶过多（触发“等量/翻新扩容”）

- **触发条件**：元素总数虽然没有达到上限，但是溢出桶（Overflow Buckets）的数量太多了。具体判定标准为：
  - 如果主桶总数 $< 2^{15}$（即 32768）：当溢出桶总数 $\ge$ 主桶总数时。
  - 如果主桶总数 $\ge 2^{15}$：当溢出桶总数 $\ge 2^{15}$ 时。
- **通俗理解**：这种情况通常发生在“频繁插入，又频繁删除”的场景。map 里的元素总量看起来并不多（装载因子没达标），但由于之前的剧烈操作，底层遗留了大量的空洞和溢出桶。
- **底层考量**：溢出桶太多会导致内存排列极度稀疏。当 Go 去查询一个不存在的 Key 时，必须顺着长长的溢出桶链条一路找下去，白白浪费 CPU 时间，同时还霸占着大量零碎的内存无法释放。
- **扩容结果**：Go 会开辟一个和原先一模一样大的新桶数组。这次扩容不增加容量，纯粹是为了把散落四处的数据重新整理、排紧凑，把不必要的溢出桶给“脱水”裁撤掉，让 map 恢复紧凑的高性能状态。因此这也被称为“等量翻新”。

> [!IMPORTANT]
> **是谁按下了扩容的开关？**
> 必须由 **【写操作】**（如 `m[k] = v` 或 `delete`）来触发。如果一个 map 已经达到了扩容的临界点，只要你一直只是 `for range` 读取它，它就永远维持现状，不会有任何动作。只有当某一个 Goroutine 试图往里面写入、修改或删除数据，调用底层 `mapassign` / `mapdelete` 函数时，Go 才会顺便看一眼判定条件，然后按下扩容的开关。

### 18. map 的扩容策略是什么？

> [!TIP]
> **核心回答：**
> Go 语言中 map 的扩容策略可以总结为：**增量搬迁，渐进扩容**。
> 当满足了扩容条件后，Go 运行时（Runtime）并不会像常规数组那样立刻一次性搬迁所有数据，而是采用了一种“延时、分批”策略，以避免整个后端的响应时间因为扩容而出现剧烈抖动（即避免性能毛刺）。

#### 一、 核心扩容策略：渐进式搬迁（Incremental Evacuation）

当 map 的写操作触发了扩容条件时，Go 只是在内存中做好了准备工作，真正的搬家是分批进行的。

1. **准备阶段（新旧交替）**
   - `hmap.oldbuckets` 指针指向当前的桶数组（变成旧桶）。
   - `hmap.buckets` 指针指向一块全新申请的桶数组空间（如果是翻倍扩容，新空间大小是旧空间的 2 倍）。
   - `hmap.nevacuate`（搬迁进度计数器）置为 0，表示刚开始，还没有旧桶被搬完。
   - `mapextra.oldoverflow` 临时接管所有的旧溢出桶。

2. **搬迁阶段（捎带脚搬家）**
   - 在扩容期间，每一次对该 map 的**写操作（赋值、修改）**或**删除（delete）**操作，都会触发一次搬迁。
   - **搬迁的数量**：每次操作，Go 会顺手搬迁 1~2 个旧桶。
     - **第一桶**：当前操作的 Key 落在哪个旧桶，Go 就优先把这个对应的旧桶搬完。
     - **第二桶**：顺便把 `nevacuate` 计数器当前指向的那个旧桶（顺位旧桶）也顺手搬了，以此保证即使你一直在操作同一个桶，整个 map 的搬迁进度也能往前推进。
   - **只读不搬**：如果在扩容期间你只是进行 `v := m[k]` 的读取操作，Go 是绝对不会帮你搬迁数据的，只会根据 `nevacuate` 的进度，去旧桶或新桶里查找。

#### 二、 数据的分流与重新排列（Rehash 规则）

当一个旧桶被选到进行搬迁时，里面的 8 个键值对以及挂在它后面的溢出桶数据，会根据扩容的类型被重新洗牌：

- **策略 A：翻倍扩容中的“细胞分裂”**
  因为新桶的数量是旧桶的 2 倍（$B$ 变成了 $B+1$），原本落在同一个旧桶里的元素，在新桶里面临二选一分流。分流的依据取决于 Key 哈希值的第 $B+1$ 位是 0 还是 1：
  - 如果第 $B+1$ 位是 0：说明这个 Key 留在低位，它会被搬迁到新桶中索引和旧桶一模一样的位置（简称 Low 位）。
  - 如果第 $B+1$ 位是 1：说明它需要去高位，它会被搬迁到新桶中 `旧桶索引 + 旧桶总数` 的位置（简称 High 位）。

- **策略 B：等量扩容中的“脱水紧凑”**
  如果是由于溢出桶过多触发的等量扩容，由于新桶和旧桶的数量完全一样（$B$ 没变）：
  - 所有的 Key 重新计算哈希后，原封不动地落回原来编号相同的桶里。
  - **核心改变**：搬迁过程中，旧桶里那些因为历史 `delete` 操作留下的“内存空洞”会被彻底忽略。原本散落在 1 个主桶 + 多个溢出桶里的零星元素，会被重新紧凑地排满在新主桶里。溢出桶数量大幅减少，完成“脱水”瘦身。

#### 三、 搬迁结束（新王登基）

随着用户的读写请求不断涌入，旧桶被一个接一个地搬空。当 `hmap.nevacuate` 的数量等于旧桶的总数时，意味着老数据交接完毕：
- `hmap.oldbuckets` 指针正式赋值为 `nil`，切断与旧内存的一切联系。
- 已经变成孤岛的旧桶、旧溢出桶内存在下一次 Go 的垃圾回收（GC）扫描时，会被系统彻底回收并释放。

### 19. make 和 new 有什么区别？

> [!TIP]
> **核心回答：**
> `new` 和 `make` 都是用来分配内存的内置函数，但它们的分工和底层行为完全不同。
> - **`new`**：用来分配任意类型的基本内存，它只负责分配并清零内存，返回指向该类型的指针（`*T`）。
> - **`make`**：专门用来分配并初始化内置的三个引用类型（`slice`、`map`、`channel`），它会进行深度初始化，返回的是类型本身（`T`）。

#### 一、 核心区别速览表

| 维度 | new 函数 | make 函数 |
| :--- | :--- | :--- |
| **适用类型** | 所有类型（`int`, `string`, 结构体等） | 仅限三种引用类型：`slice`、`map`、`channel` |
| **返回对象** | 返回指向该类型的指针（如 `*int`, `*User`） | 返回已经初始化的值类型本身（如 `[]int`, `map[string]int`） |
| **内存初始化** | 不进行初始化，仅清零内存，填入该类型的默认零值 | 完成内部结构的初始化（分配底层数组、创建哈希桶、设置参数） |

#### 二、 底层原理深度对比

1. **`new(T)` 的底层行为：清零内存**
   `new` 是一个纯粹的内存分配器。当调用 `p := new(User)` 时，Go 运行时执行：
   - 在内存中申请一块刚好容纳 `User` 结构体的空间。
   - 将这块空间的所有字节清零（也就是将其中的字段设为默认的零值）。
   - 返回这块内存的绝对地址（指针）。

   ```go
   type User struct {
       Name string
       Age  int
   }

   uPtr := new(User)
   // uPtr 的类型是 *User
   // 此时 uPtr 指向的结构体内容为 {Name: "", Age: 0}
   ```

2. **`make(T, args)` 的底层行为：深度初始化**
   对于 `slice`、`map` 和 `channel`，它们在 Go 底层都不是简单的单一块内存，而是复杂的结构体（例如 `map` 底层有 `hmap` 和 `bmap`，`slice` 底层有 `SliceHeader` 和隐式数组）。
   如果用 `new` 去开辟它们，你只是得到了一个由于清零而导致内部指针指向 `nil` 的空壳子（`nil map` 或 `nil slice`），直接使用会崩溃或无法正常工作。
   而 `make` 的存在，就是为了不仅开辟管理壳子的内存，还顺便把它们底层赖以生存的“血肉”（如 `map` 的哈希桶、`slice` 的底层数组）也一并申请并绑定好。

   ```go
   // 必须用 make，才会真正帮你在底层创建好哈希桶（buckets）
   m := make(map[string]int)
   m["apple"] = 1 // 安全写操作

   // 如果作死用 new
   mPtr := new(map[string]int) // mPtr 类型是 *map[string]int
   // 此时 *mPtr 的值是 nil！
   // (*mPtr)["apple"] = 1      // ❌ 直接崩溃：panic: assignment to entry in nil map
   ```

#### 三、 经典面试代码辨析

面试官经常会拿这样一段代码来考你，问你错在哪里：
```go
func main() {
    // 题目 A
    var p *int
    *p = 10 // ❌ 报错：panic: runtime error: invalid memory address or nil pointer dereference

    // 题目 B
    var m map[string]int
    m["key"] = 1 // ❌ 报错：panic: assignment to entry in nil map
}
```

##### 应该怎么修正？
- **题目 A 的修正（使用 `new`）**：
  指针 `p` 声明时是 `nil`，不能直接解引用赋值。必须用 `new` 给它分配一块存放整数的内存空间：
  ```go
  p := new(int) // 分配内存，p 变成有效指针
  *p = 10       // 成功
  ```
- **题目 B 的修正（使用 `make`）**：
  Map 声明时底层结构没有初始化，必须用 `make` 注入灵魂：
  ```go
  m := make(map[string]int) // 初始化底层 hmap 结构
  m["key"] = 1              // 成功
  ```

> [!TIP]
> **总结口诀**
> “`New` 任意类型返回针（指针），内存清零给零值；`Make` 专给三兄弟（`Slice`, `Map`, `Channel`），深度初始化返其身（返回类型本身）。”

### 20. slice、map、channel 创建时的几个参数分别代表什么含义？

> [!TIP]
> **核心回答：**
> 使用 `make` 函数创建 Slice、Map 和 Channel 时，传入的参数代表的含义和底层行为各不相同。这些参数本质上都是在向 Go 运行时（Runtime）申请不同规格的内存空间。

#### 1. Slice（切片）的参数含义

创建切片时，`make` 最多可以接收 3 个参数：
```go
s := make([]int, length, capacity)
```
- **参数一：类型（`[]int`）**：告诉 Go 你要创建一个存放什么类型元素的切片。
- **参数二：长度（`length / len`）—— 必填**：
  - **含义**：切片当前可见且可直接通过索引读写的元素个数。
  - **底层行为**：Go 会在底层数组里为你直接初始化 `length` 个该类型的零值（如果是 `int` 就是 0）。
  - **注意**：此时你直接访问 `s[length-1]` 是安全的，但访问 `s[length]` 就会触发越界崩溃。
- **参数三：容量（`capacity / cap`）—— 选填**：
  - **含义**：底层数组总共分配的元素槽位数量，即在不触发重新分配内存扩容的情况下最多能装的数据量。
  - **缺省规则**：如果不传该参数，容量默认等于长度（即 `cap = len`）。

##### 💡 Web3 实战避坑：
如果你要在后端接收一批链上交易：
```go
// ❌ 错误写法：长度给 10，随后又 append 10 次
txs := make([]Transaction, 10)
for i := 0; i < 10; i++ { 
    txs = append(txs, fetchTx(i)) 
}
// 结果：txs 的长度变成了 20，前 10 个全都是空结构体零值！

// ✅ 正确写法 A：长度给 0，容量给 10（推荐，append 绝不触发扩容搬迁）
txs := make([]Transaction, 0, 10)
for i := 0; i < 10; i++ { 
    txs = append(txs, fetchTx(i)) 
}

// ✅ 正确写法 B：长度给 10，直接索引赋值
txs := make([]Transaction, 10)
for i := 0; i < 10; i++ { 
    txs[i] = fetchTx(i) 
}
```

#### 2. Map（映射）的参数含义

创建 Map 时，`make` 最多接收 2 个参数：
```go
m := make(map[string]int, sizeHint)
```
- **参数一：类型（`map[string]int`）**：定义键值对的类型，左边是 Key 类型（必须是可比较类型），右边是 Value 类型。
- **参数二：初始容量提示（`sizeHint`）—— 选填**：
  - **含义**：告诉 Go 运行时，这个 map 大概预计要存多少条数据。
  - **底层行为**：Go 会根据你给的数字，通过数学计算，在堆内存中一次性开辟好足够数量的哈希桶（Buckets）。
  - **缺省规则**：如果不传，Go 只会给一个极小的默认分配，后续随着数据写入不断触发渐进式扩容。

##### 💡 Web3 实战建议：
Map 没有“长度（length）”这个参数，因为它是无序散列的。如果你明确知道要从某个区块捞出 500 个代币的 Balance 状态，请务必写成 `make(map[address]uint256, 500)`，这样可以彻底免疫运行期间由于频繁触发扩容带来的哈希重排（Rehash）CPU 损耗。

#### 3. Channel（通道）的参数含义

创建 Channel 时，`make` 最多接收 2 个参数：
```go
ch := make(chan int, bufferSize)
```
- **参数一：类型与方向（`chan int`）**：定义通道里传输的数据类型。
- **参数二：缓冲区大小（`bufferSize`）—— 选填**：
  - **不传或传 0**（`make(chan int)`）：称为 **“无缓冲通道”**。发送方（`ch <- 1`）必须死死等到接收方（`<-ch`）准备好接手，否则发送方会直接阻塞卡住。这是一种强同步机制。
  - **传大于 0 的数**（`make(chan int, 10)`）：称为 **“有缓冲通道”**。通道内部自带一个长度为 10 的先进先出（FIFO）队列。发送方可以往里扔 10 个数据而不被阻塞；只有当第 11 个数据扔进去且没人取时，发送方才会阻塞。

##### 💡 并发模型实战：
在写多协程监听链上 Event 的架构时（比如一个线程扫块，多个线程处理 Event）：
- 如果追求数据绝对不错过、强步骤同步 👉 选**无缓冲通道**。
- 如果为了应对公链出块瞬间的高并发流量洪峰，给消费协程留出喘息缓冲时间 👉 选**有缓冲通道**（如 `make(chan Event, 1024)`）。

> [!TIP]
> **总结口诀**
> - Slice 的两个数：前面是“当前能用几个（len）”，后面是“底层最多能装几个（cap）”。
> - Map 的那个数：是“免扩容的装载预算（sizeHint）”。
> - Channel 的那个数：是“快递柜的格子数（bufferSize）”，格子没满，发数据就不需要排队。

### 21. 线程安全的 map 如何实现？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，实现线程安全（并发安全）的 map 主要有以下四种工业级方案。根据你的具体业务场景（比如是读多写少，还是高并发巨量读写），我们需要选择不同的实现策略。

#### 方案一：普通 Map + 读写锁 `sync.RWMutex`（最通用、首选）

这是最经典、最容易维护的方案。利用标准库的 `sync.RWMutex`，让多个协程可以同时读取，但同一时间只能有一个协程写入。

##### 💻 代码实现：
```go
package main

import "sync"

type RWBlockCache struct {
	mu    sync.RWMutex
	cache map[uint64]string // 假设缓存区块高度到 Hash 的映射
}

func NewRWBlockCache() *RWBlockCache {
	return &RWBlockCache{
		cache: make(map[uint64]string),
	}
}

// Read: 允许多个协程同时读
func (c *RWBlockCache) Get(key uint64) string {
	c.mu.RLock()         // 加读锁
	defer c.mu.RUnlock() // 释放读锁
	return c.cache[key]
}

// Write: 排他写，写的时候别人既不能读也不能写
func (c *RWBlockCache) Set(key uint64, val string) {
	c.mu.Lock()         // 加写锁
	defer c.mu.Unlock() // 释放写锁
	c.cache[key] = val
}
```
- **适用场景**：大多数常规业务，对读写比例没有极端要求的场景。
- **优缺点**：逻辑简单安全；但在极高并发写的情况下，写锁会导致大量协程排队挂起，产生锁竞争。

#### 方案二：官方自带的 `sync.Map`（适合“读多写少”的极端场景）

Go 官方标准库提供了一个专门应对高并发的 `sync.Map`。它不需要你手动加锁，内部采用了读写分离、原子操作（Atomic）以及底层双 map 快照机制（一个只读的 `read map`，一个负责写入的 `dirty map`）。

##### 💻 代码实现：
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var m sync.Map

	// 1. 写入 (Store)
	m.Store("eth_price", 2500)
	m.Store("btc_price", 60000)

	// 2. 读取 (Load)
	if val, ok := m.Load("eth_price"); ok {
		fmt.Println("ETH 价格:", val.(int)) // 注意：返回的是 interface{}，需要类型断言
	}

	// 3. 存在则读取，不存在则写入 (LoadOrStore)
	actual, loaded := m.LoadOrStore("sui_price", 5)
	fmt.Printf("实际值: %v, 是否是读取到的: %v\n", actual, loaded)

	// 4. 删除 (Delete)
	m.Delete("btc_price")

	// 5. 遍历 (Range)
	m.Range(func(key, value interface{}) bool {
		fmt.Printf("Key: %v, Value: %v\n", key, value)
		return true // 返回 true 继续迭代，false 停止
	})
}
```
- **适用场景**：读极多、写极少、Key 相对稳定的场景。例如：缓存系统的配置项、系统的白名单、节点网关中长连接的路由表。
- **致命缺点**：如果你的业务是“高频写入/删除”，`sync.Map` 内部会频繁触发从 `dirty` 到 `read` 的内存同步和提升复制，性能会雪崩，甚至远不如普通加锁的 map。

#### 方案三：分段锁（Sharded Map / Map 撕裂）（适合高并发巨量读写）

如果你的后端需要同时处理成千上万个用户的钱包状态读写，方案一的锁粒度就太粗了（所有人抢一把锁）。此时工业界的标准做法是分段锁（分片锁）。
- **运作逻辑**：把一个巨大的 map 拆分成 32 个或 64 个小 map（分片），每个小 map 拥有一把独立的锁。当请求一个 Key 时，先通过哈希函数计算出这个 Key 的哈希值，再取余（例如 `Hash(key) % 32 = 4`），就去抢对应分片的微型锁。
- **推荐开源库**：`github.com/orcaman/concurrent-map`
- **适用场景**：超高并发的巨量读写，能将锁竞争（Lock Contention）降低数十倍。

#### 方案四：用 Channel 构建守护协程（完全无锁、Actor 模型）

如果限制不能使用任何锁，也不允许用 `sync.Map`，可以使用 Channel 方案。将普通 map 锁在唯一的一个守护协程内，所有外部读写全部通过 Channel 变成排队的“消息”。因为始终只有这一个协程可以碰 map，从根本上消灭了数据竞争。

#### 💡 生产环境选型终极指南（Web3 后端视角）

1. **写多链状态监控、区块数据同步（高频写，高频读）**：
   👉 选 **方案三（分段锁 / concurrent-map）**。因为区块出块或者扫块时流量是暴增的，分段锁能完美平摊并发压力。
2. **存动态白名单、DApp 全局固定配置（极少写，全员读）**：
   👉 选 **方案二（sync.Map）**。利用只读快照机制，读操作可以实现零锁阻塞。
3. **常规后台、单体 API 接口（普通读写）**：
   👉 选 **方案一（普通 map + sync.RWMutex）**。代码最简单，最不容易写出逻辑 bug。

### 22. Go 结构体（struct）能不能比较？

> [!TIP]
> **核心回答：**
> 答案是：**取决于结构体内部包含的字段类型**。
> - 如果一个结构体内部的所有字段都是**可比较的**，那么这个结构体就是可比较的。
> - 如果结构体中包含任何**不可比较的字段**（如 `slice`、`map`、`func`），它就不能直接用 `==` 或 `!=` 进行比较。

#### 一、 哪些结构体可以比较？

如果结构体的所有字段都属于基础类型（如 `int`, `string`, `float`, `bool`）或数组、指针，那么它们可以直接使用 `==` 和 `!=` 进行比较。比较时，会按字段顺序依次对比值是否完全相等。

##### 💻 代码示例
```go
package main

import "fmt"

type Point struct {
	X int
	Y int
}

func main() {
	p1 := Point{X: 1, Y: 2}
	p2 := Point{X: 1, Y: 2}
	p3 := Point{X: 3, Y: 4}

	fmt.Println(p1 == p2) // 输出: true  (字段值完全相同)
	fmt.Println(p1 != p3) // 输出: true  (值不同)
}
```

> [!WARNING]
> 即使两个结构体的字段完全一样，如果它们类型名字不同，也不能直接比较，必须先进行显式类型转换。
> ```go
> type Position Point // Position 是新类型
> pos := Position{X: 1, Y: 2}
> // fmt.Println(p1 == pos) // ❌ 编译报错：类型不匹配
> fmt.Println(p1 == Point(pos)) // ✅ 转换后可比，输出 true
> ```

#### 二、 哪些结构体不能比较？

根据 Go 语言规范，有三种类型是绝对不可比较的（除了能和 `nil` 比较外，它们之间不能用 `==`）：
- **`slice`**（切片）
- **`map`**（映射）
- **`func`**（函数类型）

如果你的结构体里包含了这三者中的任意一个，这个结构体在编译时就会被直接打上“不可比较”的烙印。

##### ❌ 错误示范（编译报错）：
```go
package main

type TxCache struct {
	BlockNumber uint64
	TxHashes    []string // ❌ 包含了 slice，导致整个结构体不可比！
}

func main() {
	t1 := TxCache{BlockNumber: 100, TxHashes: []string{"0x123"}}
	t2 := TxCache{BlockNumber: 100, TxHashes: []string{"0x123"}}

	// fmt.Println(t1 == t2)
	// ❌ 编译直接报错: invalid operation: t1 == t2 (struct containing []string cannot be compared)
}
```

#### 三、 包含不可比字段的结构体，怎么强行比较？

如果必须比对它们是否相等，有以下三种工业级解决方案：

1. **方案 1：使用标准库 `reflect.DeepEqual`（最通用但性能低）**
   `reflect` 包提供的 `DeepEqual` 可以进行深度递归比较。它会无视“不可比类型”的限制，逐层深入扫描所有指针、切片、字典内部的具体值。
   ```go
   package main

   import (
       "fmt"
       "reflect"
   )

   type TxCache struct {
       BlockNumber uint64
       TxHashes    []string
   }

   func main() {
       t1 := TxCache{BlockNumber: 100, TxHashes: []string{"0x123"}}
       t2 := TxCache{BlockNumber: 100, TxHashes: []string{"0x123"}}

       // 使用反射进行深比较
       fmt.Println(reflect.DeepEqual(t1, t2)) // 输出: true
   }
   ```
   - **代价**：反射（Reflection）非常笨重。它在运行时动态解析类型，如果在高频循环或扫块数据解析中使用，会导致 CPU 开销暴增。

2. **方案 2：手写 `Equal` 方法（极致性能、生产环境标准做法）**
   如果你追求极致性能，最好的做法是为结构体显式编写一个 `Equal` 方法，手动去比对基础字段，并循环比对切片。
   ```go
   func (t *TxCache) Equal(other *TxCache) bool {
       if t.BlockNumber != other.BlockNumber {
           return false
       }
       if len(t.TxHashes) != len(other.TxHashes) {
           return false
       }
       for i := range t.TxHashes {
           if t.TxHashes[i] != other.TxHashes[i] {
               return false
           }
       }
       return true
   }
   ```
   - **优点**：性能最高，比反射快几十倍，且没有任何额外内存分配。

3. **方案 3：利用 Google 开源的 `go-cmp` 库（自动化测试利器）**
   在写单元测试时，如果手写 `Equal` 觉得太麻烦，可以用 Google 官方维护的 `github.com/google/go-cmp/cmp` 库，它比 `reflect.DeepEqual` 更安全、更强大，且能清晰打印出是哪一个字段不一致。
   ```go
   import "github.com/google/go-cmp/cmp"

   if !cmp.Equal(t1, t2) {
       // 它不仅能比对，还能用 cmp.Diff 打印出不一致的差异在哪里
       fmt.Println(cmp.Diff(t1, t2))
   }
   ```

##### 💡 隐藏彩蛋：结构体能不能当 Map 的 Key？
问：“自定义的 struct 能不能作为 map 的 key，比如 `map[User]bool`？”
答：“只有**可比较的结构体**才能作为 map 的 key。如果结构体包含 slice 或 map，将其作为 key 会在编译时直接报错。”

### 23. map 如何顺序读取？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，map 的底层是无序的。为了防止开发者依赖当前的遍历顺序，Go 在底层源码中做了一个故意为之的设计：每次使用 `for range` 遍历 map 时，底层都会引入一个随机种子，故意打乱遍历的起点。
> 想要顺序读取 map，标准的工业级解决方案有以下三种：
> 1. **提取 Key 排序法**（最通用、临时用用首选）
> 2. **结构体切片代替法**（适合写入顺序读取）
> 3. **双轨制组合结构**（适合高频 $O(1)$ 查找 + 顺序遍历）

#### 方案一：提取 Key 排序法（最经典、最常用）

- **核心思想**：先把 map 所有的 Key 提取出来，放到一个切片里，对这个切片进行排序，接着遍历排序后的切片，拿切片里的 Key 去 map 中依次读取 Value。

##### 💻 代码实现（Go 1.21+ 泛型写法）：
```go
package main

import (
	"fmt"
	"maps"
	"slices"
)

func main() {
	// 假设这是一个存储代币符号和其排名的 map（天生无序）
	tokenRank := map[string]int{
		"ETH": 2,
		"BTC": 1,
		"SUI": 3,
		"SOL": 4,
	}

	// 1. 提取所有的 Key（使用 Go 1.21 新增的 maps.Keys 泛型函数）
	keys := slices.Collect(maps.Keys(tokenRank))

	// 2. 对 Key 切片进行排序（按字母表 A-Z 排序）
	slices.Sort(keys)

	// 3. 顺序读取
	for _, k := range keys {
		fmt.Printf("Token: %s, Rank: %d\n", k, tokenRank[k])
	}
}
```
- **输出结果**（无论运行多少次，顺序都恒定不变）：
  ```plaintext
  Token: BTC, Rank: 1
  Token: ETH, Rank: 2
  Token: SOL, Rank: 4
  Token: SUI, Rank: 3
  ```
- **复杂度分析**：排序的开销为 $O(n \log n)$。如果你的 map 尺寸非常大，且需要频繁按顺序读取，每次读取都排一次序会非常消耗 CPU。

#### 方案二：结构体切片代替法（适合“按写入顺序”读取）

如果你的需求并不是要按 Key 的大小排序，而是“希望按照写入 map 的先后顺序读取出来”（即 FIFO 队列特性），最优雅的架构设计其实是直接使用结构体切片。

##### 💻 代码实现：
```go
package main

import "fmt"

type TokenInfo struct {
	Symbol string
	Rank   int
}

func main() {
	// 按照写入顺序，直接排列在切片里，切片天生有序
	orderedTokens := []TokenInfo{
		{Symbol: "BTC", Rank: 1},
		{Symbol: "ETH", Rank: 2},
		{Symbol: "SUI", Rank: 3},
	}

	// 顺序读取极其简单，性能为 O(n)
	for _, info := range orderedTokens {
		fmt.Printf("Token: %s, Rank: %d\n", info.Symbol, info.Rank)
	}
}
```

#### 方案三：双轨制组合结构（适合高频 O(1) 查找 + 顺序遍历）

很多高级分布式系统中，业务既要求能根据 Key 在 $O(1)$ 时间内快速查找/修改，又要求能随时按顺序遍历。此时，标准的做法是将 Map 和 Slice 组合成一个自定义类型，维护一个“双轨制”结构：

##### 💻 代码实现：
```go
package main

import "fmt"

type OrderedMap struct {
	keys []string       // 专门负责记录顺序
	data map[string]int // 专门负责快速查找
}

func NewOrderedMap() *OrderedMap {
	return &OrderedMap{
		keys: make([]string, 0),
		data: make(map[string]int),
	}
}

// 写入：同时写入 map 和 slice
func (om *OrderedMap) Set(key string, val int) {
	if _, exists := om.data[key]; !exists {
		om.keys = append(om.keys, key) // 只有新 key 才记录顺序
	}
	om.data[key] = val
}

// 查找：O(1) 极速响应
func (om *OrderedMap) Get(key string) (int, bool) {
	val, ok := om.data[key]
	return val, ok
}

// 顺序遍历
func (om *OrderedMap) Range() {
	for _, k := range om.keys {
		fmt.Printf("Key: %s, Value: %v\n", k, om.data[k])
	}
}

func main() {
	om := NewOrderedMap()
	om.Set("First", 10)
	om.Set("Second", 20)
	om.Set("Third", 30)

	om.Range() // 严格按照 "First" -> "Second" -> "Third" 的写入顺序输出
}
```
- **优点**：完美兼顾了 Map 的快速查找和 Slice 的顺序保持。
- **缺点**：内存开销翻倍；且在发生 `delete` 操作时，切片也需要跟着同步剔除元素（会涉及到切片元素的移动，产生 $O(n)$ 的开销）。

### 24. Go 中如何实现 Set？

> [!TIP]
> **核心回答：**
> Go 官方标准库并没有直接提供 Set 集合结构。在工业界，最标准、最高效的做法是**利用 map 的 Key 具有唯一性（不可重复）这一底层特性，来封装并模拟出一个 Set**。

#### 方案一：利用空结构体 `struct{}` 实现极致性能的单体 Set

- **核心优化点**：用 `struct{}` 替代 `bool` 作为 Value。在 Go 中，`bool` 变量占用 1 字节，而空结构体 `struct{}` 占用 **0 字节** 内存。在底层桶中，`struct{}` 完全不占用任何内存空间，能实现零内存占位。

##### 💻 代码实现（基于 Go 泛型）：
```go
package main

import "fmt"

// 使用泛型限制 Key 必须是可比较类型 (comparable)
type Set[T comparable] struct {
	m map[T]struct{}
}

// 初始化 Set
func NewSet[T comparable]() *Set[T] {
	return &Set[T]{m: make(map[T]struct{})}
}

// 添加元素（自动去重）
func (s *Set[T]) Add(item T) {
	s.m[item] = struct{}{} // 填充空结构体占位
}

// 删除元素
func (s *Set[T]) Remove(item T) {
	delete(s.m, item)
}

// 检查是否包含某个元素
func (s *Set[T]) Contains(item T) bool {
	_, exists := s.m[item]
	return exists
}

// 获取 Set 大小
func (s *Set[T]) Size() int {
	return len(s.m)
}

func main() {
	// 创建一个专门存以太坊地址（String）的 Set
	addressSet := NewSet[string]()
	addressSet.Add("0x123")
	addressSet.Add("0x456")
	addressSet.Add("0x123") // 重复添加，底层会自动覆盖去重

	fmt.Println("Set 大小:", addressSet.Size())       // 输出: 2
	fmt.Println("包含 0x123:", addressSet.Contains("0x123")) // 输出: true
}
```

#### 方案二：工业级并发安全的 Thread-Safe Set

如果需要在多协程并发操作 Set，为了避免并发写 map 触发崩溃，需要加上读写锁 `sync.RWMutex`：

##### 💻 代码实现：
```go
package main

import "sync"

type ConcurrentSet[T comparable] struct {
	mu sync.RWMutex
	m  map[T]struct{}
}

func NewConcurrentSet[T comparable]() *ConcurrentSet[T] {
	return &ConcurrentSet[T]{m: make(map[T]struct{})}
}

func (s *ConcurrentSet[T]) Add(item T) {
	s.mu.Lock() // 加写锁
	defer s.mu.Unlock()
	s.m[item] = struct{}{}
}

func (s *ConcurrentSet[T]) Contains(item T) bool {
	s.mu.RLock() // 加读锁，允许多个协程同时判断 Contains，性能极高
	defer s.mu.RUnlock()
	_, exists := s.m[item]
	return exists
}

func (s *ConcurrentSet[T]) Remove(item T) {
	s.mu.Lock()
	defer s.mu.Unlock()
	delete(s.m, item)
}
```

#### 方案三：开源成熟解方案

在实际的生产项目中，如果要进行复杂的数学运算（如交集、并集、差集等），通常直接引入第三方库：
👉 `github.com/deckarep/golang-set/v2`

##### 💻 代码调用：
```go
package main

import (
	"fmt"
	mapset "github.com/deckarep/golang-set/v2"
)

func main() {
	// 创建一个非线程安全的 Set（如果需要安全的，可以调用 NewThreadSafeSet）
	setA := mapset.NewSet[int](1, 2, 3)
	setB := mapset.NewSet[int](3, 4, 5)

	// 1. 求交集 (Intersection)
	intersect := setA.Intersect(setB)
	fmt.Println("交集:", intersect) // 输出: [3]

	// 2. 求并集 (Union)
	union := setA.Union(setB)
	fmt.Println("并集:", union) // 输出: [1, 2, 3, 4, 5]
}
```

### 25. 使用值为 nil 的 slice、map 会发生什么？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，一个只声明但没有初始化的切片（Slice）和字典（Map），它们的默认值都是 `nil`。
> - **`nil` 切片**：非常安全，几乎可以像正常切片一样随便用（比如 `append`、`len`、`range`），唯一的雷区是**直接通过索引去写数据**。
> - **`nil` map**：非常危险，进行**写入操作会直接导致程序崩溃（Panic）**；不过读取、删除、求长度都是安全的。

#### 一、 使用值为 nil 的 Slice 会发生什么？

`nil` 切片在底层的三个字段为：指针 `ptr = nil`、长度 `len = 0`、容量 `cap = 0`。
1. **进行【读操作 / 获取长度 / 遍历】 👉 绝对安全**
   调用 `len(s)`、`cap(s)`，或用 `for range` 去遍历它，程序都能安全运行，完全不会报错，返回值为 0 或直接跳过。
2. **进行【追加操作（`append`）】 👉 绝对安全，自动初始化**
   可以直接对 `nil` 切片执行 `append`。此时 Go 会自动分配底层数组并绑定。
   ```go
   var s []int
   s = append(s, 100) // 安全！
   fmt.Println(s)     // 输出: [100]
   ```
3. **唯一的雷区：【直接通过索引写数据】 👉 崩溃（Panic）**
   由于长度 `len` 为 0，如果直接用下标索引去写入或访问，会触发越界：
   ```go
   var s []int
   s[0] = 100 // ❌ 报错崩溃：panic: runtime error: index out of range [0] with length 0
   ```

#### 二、 使用值为 nil 的 Map 会发生什么？

`nil` map 在底层只有栈上的一个指针指向 `nil`（`0x0`），哈希表结构 `hmap` 完全没有被创建。
1. **进行【写操作 / 插入 / 修改】 👉 直接崩溃（致命 Panic）**
   这是最容易踩坑的知识点：
   ```go
   var m map[string]int
   m["eth"] = 2000 // ❌ 报错崩溃：panic: assignment to entry in nil map
   ```
2. **进行【读操作 / 删除操作 / 求长度】 👉 绝对安全**
   除了写之外，其他操作在 `nil` map 上都是被安全允许的：
   - 读操作：找不到就安全地返回该类型的“零值”。
   - 删除操作：直接被编译器忽略，无事发生。
   - 获取长度：返回 0。

#### 三、 核心总结与避坑表格

| 操作行为 | 对 nil Slice (`var s []int`) | 对 nil Map (`var m map[k]v`) |
| :--- | :--- | :--- |
| **`== nil` 判断** | `true` | `true` |
| **读取长度 `len()`** | 安全，返回 0 | 安全，返回 0 |
| **读取元素（索引/键）** | ❌ 崩溃（`index out of range`） | 安全，返回 Value 类型的默认零值 |
| **直接写入元素** | ❌ 崩溃（`index out of range`） | ❌ 崩溃（`assignment to entry in nil map`） |
| **内置追加/删除** | 安全（`append` 会自动分配内存） | 安全（`delete` 会直接被忽略） |
| **`for range` 遍历** | 安全，直接跳过不执行循环体 | 安全，直接跳过不执行循环体 |

> [!NOTE]
> **生产环境避坑建议**
> - **对于 Map**：声明时千万不要只写 `var m map[k]v` 然后直接用。要么用字面量直接初始化：`m := map[string]int{}`，要么用标准 `make` 创建：`m := make(map[string]int)`。
> - **对于 Slice**：在编写 Gin 框架的 DApp 链下 API 时，如果向前端返回一个空的列表，`nil` 切片在 JSON 序列化后会变成 `null`，容易导致前端网页报错。因此，建议一律初始化为空切片：`s := []int{}`，这样 JSON 序列化后会得到符合前端预期的空数组 `[]`。
