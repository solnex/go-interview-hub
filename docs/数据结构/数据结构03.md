### 26. Golang 有没有 this 指针？

> [!TIP]
> **核心回答：**
> Go 语言中**没有**类似于 Java 或 C++ 那样的 `this` 或 `self` 关键字。
> Go 是通过在方法定义中声明**“接收者（Receiver）”**机制，完美实现了面向对象中 `this` 指针的功能。

#### 一、 什么是接收者（Receiver）？

在 Go 语言中，如果你想给一个结构体（struct）绑定一个方法，你需要把变量明确地写在函数名的前面。这个写在前面的变量就叫做接收者。你可以自由地为这个接收者命名，业内最标准的做法是使用结构体类型名称的首字母小写。

##### 💻 代码示例：
```go
package main

import "fmt"

type Wallet struct {
	Address string
	Balance uint64
}

// 这里的 (w *Wallet) 就是接收者，w 就相当于其他语言中的 this
func (w *Wallet) ShowInfo() {
	// 直接通过 w 访问结构体内部的属性
	fmt.Printf("钱包地址: %s, 余额: %d\n", w.Address, w.Balance)
}

func main() {
	myWallet := &Wallet{Address: "0x123...", Balance: 100}
	// 调用方法，Go 会自动把 myWallet 的指针传给方法内部的 w
	myWallet.ShowInfo()
}
```

#### 二、 Go 为什么不用 this 关键字？

Go 语言的设计哲学是“显式优于隐式”。去掉 `this` 主要有以下两个极大的好处：

1. **明确你是“值拷贝”还是“指针引用”**
   在 Java/C++ 中，`this` 永远是指向当前对象的指针。但在 Go 中，面向对象的方法支持两种接收者模式，它们在内存中的行为完全不同：
   - **指针接收者（`*Wallet`）**：类似于传统的 `this` 指针。方法内部拿到的是原对象的内存地址，在方法里修改属性，会直接影响到外部原对象。
   - **值接收者（`Wallet`）**：方法内部拿到的是原对象的一个完整副本（值拷贝）。在方法里无论怎么修改属性，外部原对象都毫无影响。

   ```go
   // 模式 A：指针接收者（相当于 this 指针）
   func (w *Wallet) DepositPointer(amount uint64) {
       w.Balance += amount // 外部的 myWallet.Balance 真的变了
   }

   // 模式 B：值接收者（纯副本，传统语言做不到）
   func (w Wallet) DepositValue(amount uint64) {
       w.Balance += amount // ❌ 修改的是副本，外部的 myWallet.Balance 毫无变化
   }
   ```
   通过自定义接收者名字（如 `w`）并显式写出 `*` 号，开发者一眼就能看懂该方法是否会修改外部的状态变量。

2. **代码可读性更高，避免命名冲突**
   如果方法内部的局部变量、或者传入的参数刚好和结构体的属性同名，在传统的带有 `this` 的语言里，你必须写成 `this.name = name`。而在 Go 中，因为接收者的名字是你自己定义的（比如 `w`），你可以非常清晰地写成 `w.Name = name`，代码极具辨识度，完全不会产生混淆。

#### 三、 工业界编写规范

根据 Go 官方的 `CodeReviewComments` 规范：
> 接收者的命名应该简短（通常是 1 到 2 个字母），并且首选类型名称的首字母小写（例如 `Wallet` 的接收者用 `w`，`Block` 的接收者用 `b`）。不要使用诸如 `this` 或 `self` 这种面向对象语言的通用隐式名称。

### 27. Go 的接收者（Receiver）机制与 C# 的什么机制类似？

> [!TIP]
> **核心回答：**
> Go 语言中通过“显式指定接收者”来为既有类型绑定方法的机制，与 C# 中的**扩展方法（Extension Methods）**机制在设计理念和最终的使用体验上最为相似。

#### 一、 为什么说它像 C# 的扩展方法？

在 C# 中，如果你想给一个普通的结构体或者类添加一个新方法，但你又不想或者不能去修改它原本的源代码，你会写一个静态方法，并在第一个参数前加上 `this` 关键字。

##### 💻 C# 扩展方法语法示例：
```csharp
public static class WalletExtensions
{
    // 这里的 (this Wallet w) 就是 C# 的扩展方法语法
    public static void ShowInfo(this Wallet w)
    {
        Console.WriteLine($"地址: {w.Address}, 余额: {w.Balance}");
    }
}
```

##### 💻 Go 语言接收者语法示例：
```go
// 这里的 (w *Wallet) 是 Go 的接收者语法
func (w *Wallet) ShowInfo() {
    fmt.Printf("地址: %s, 余额: %d\n", w.Address, w.Balance)
}
```

##### 核心相似点：
- **调用方式完全一样**：在外部调用时，都是直接用 `实例对象.方法名()`（例如 `myWallet.ShowInfo()`）。
- **底层本质都是语法糖**：
  - C# 的编译器在编译时，会把 `myWallet.ShowInfo()` 自动还原成静态调用：`WalletExtensions.ShowInfo(myWallet)`。
  - Go 语言在编译时，同样会把接收者方法还原成普通的函数调用，把对象作为第一个参数传进去。

#### 二、 它们之间的关键区别（Go 更彻底）

- **Go 没有“类内部”和“类外部”的区别**：
  - C# 中的基础方法必须写在 `class Wallet { ... }` 内部（此时里面用隐式的 `this`），只有扩展方法才写在外面。
  - Go 语言的结构体内部只能定义属性字段，所有的方法全都是在结构体外面通过接收者绑定上去的。
- **访问权限差异**：
  - C# 扩展方法由于写在外部静态类里，它无法访问被标记为 `private`（私有）的字段。
  - Go 中只要接收者方法和结构体定义在同一个包（package）里，就可以直接访问该结构体里的任何私有字段（首字母小写的字段）。
- **对值类型修改的支持**：
  - C# 扩展方法如果操作的是 `struct`（值类型），默认是值拷贝，无法直接修改原对象，必须配合特殊的 `this ref Wallet w` 语法。
  - Go 直接在语法层面上让你选择：用 `(w Wallet)` 就是值拷贝，用 `(w *Wallet)` 就是指针引用。

### 28. Go 语言中局部变量和全局变量的缺省值是什么？

> [!TIP]
> **核心回答：**
> Go 语言中不存在“未初始化”的垃圾内存。所有变量在声明时，如果没有显式赋值，Go 都会自动为其分配该类型的**“零值（Zero Value）”**。无论是全局变量（包级变量）还是局部变量，其缺省值完全取决于它们的数据类型。

#### 一、 各数据类型的缺省值（零值对照表）

| 类型大类 | 具体数据类型 | 缺省值（零值） | 直观表现 |
| :--- | :--- | :--- | :--- |
| **布尔型** | `bool` | `false` | `false` |
| **整型** | `int`, `int64`, `uint`, `byte` | `0` | `0` |
| **浮点型** | `float32`, `float64` | `0.0` | `0` |
| **复数型** | `complex64`, `complex128` | `0+0i` | `(0+0i)` |
| **字符串** | `string` | `""` | 空字符串，注意不是 `nil` |
| **指针** | `*int`, `*struct` | `nil` | 空指针 |
| **复合容器** | `slice`, `map`, `chan` | `nil` | 底层管理结构未初始化的空状态 |
| **接口** | `interface{}`, `any` | `nil` | 动态类型和动态值均为 `nil` |
| **函数** | `func()` | `nil` | 未绑定的空函数 |

#### 二、 复合类型（数组与结构体）的缺省规则

对于数组和结构体，Go 的规则是“递归清零”。也就是说，它们里面的每一个成员、每一个槽位，都会被自动填入对应类型的零值。

1. **数组（Array）的缺省值**：
   ```go
   var arr [3]int     // 结果为: [0, 0, 0]
   var strArr [2]string // 结果为: ["", ""]
   ```
2. **结构体（Struct）的缺省值**：
   ```go
   type Transaction struct {
       Nonce    uint64
       To       string
       IsActive bool
   }

   var tx Transaction
   // tx.Nonce    -> 0
   // tx.To       -> ""
   // tx.IsActive -> false
   ```

#### 三、 局部变量与全局变量在“语法和生命周期”上的微妙差异

- **全局变量（包级别变量）：必须使用 `var`**
  全局变量在编译期就确定了内存分配，支持显式或隐式地赋予零值。
  ```go
  var globalInt int // 有效！隐式初始化为 0
  ```
- **局部变量：必须被使用**
  在函数内部，我们通常喜欢用短变量声明 `:=`（如 `count := 0`），但是短变量声明必须显式赋初值，无法用来声明缺省值。若想让局部变量拥有缺省值，必须依然使用 `var` 关键字。
  ```go
  func doSomething() {
      var localVar int // 自动初始化为 0
      fmt.Println(localVar)
      
      // 💡 编译器硬性限制：局部变量如果声明了但后续没被使用，编译直接报错！
      // （全局变量声明了不用则不会报错）
  }
  ```

### 29. Go 语言中的“值传递”是什么意思？

> [!IMPORTANT]
> **核心回答：**
> “值传递”（Pass by Value）是指：当你在调用函数、把一个变量作为参数传进去时，系统并不会把这个变量“本体”直接传过去，而是在内存中**复制一份一模一样的副本（Copy）**传给函数。
> **Go 语言中只有值传递，没有引用传递。**

#### 一、 基础类型的“值传递”

对于整型、布尔型、字符串、结构体等基本类型，值传递的表现非常简单直接：
```go
package main

import "fmt"

func modifyValue(x int) {
	x = 999 // 在函数内部修改副本 x
}

func main() {
	num := 10
	modifyValue(num) // 把 num 传进去（此时发生了内存拷贝，复制了一个 10）
	fmt.Println(num) // 输出: 10 (外部的 num 根本没变)
}
```

#### 二、 为什么修改 Map 和指针，外面会变？

既然 Go 只有值传递，那为什么把一个 `map` 或者指针传给函数，在函数里修改它，外面的变量也会变？
> **答案是**：因为值传递拷贝的是“钥匙（指针/地址）”，而不是“房子（数据）”。

1. **指针传参的本质**：
   当你把一个指针 `p` 传给函数时，Go 复制了一份全新的指针副本。虽然指针是新的，但它内部记录的内存地址和外部的旧指针一模一样。函数内部通过地址去访问并修改了同一块物理内存上的值，所以外面也能看到。
   ```go
   func modifyPointer(p *int) {
       *p = 999 // 通过地址副本，修改了同一块物理内存上的值
   }
   ```
2. **切片（Slice）传参的本质**：
   切片本质上是一个包含 3 个字段的 24 字节结构体：`[指针, len, cap]`。
   当你把切片传给函数时，Go 复制了一份全新的 24 字节结构体副本传进去。
   - **现象 A（改元素）**：因为副本结构体里的“指针”依然指向同一个底层数组，所以你在函数内修改 `s[0] = 666`，外部可见。
   - **现象 B（扩容/追加）**：如果在函数内执行 `s = append(s, 4)`，由于发生了值传递，函数只修改了它自己手里那份副本的 `len`。函数退出后，外部切片的 `len` 依然是原来的值，所以外部完全看不到新追加的 4。

> [!NOTE]
> 在 Go 语言中，“值传递”是一条不可动摇的铁律。不要被“引用类型”这个名字骗了。在 Go 中，传递任何东西都会发生一次内存拷贝。区别只在于：
> - 传普通变量：拷贝的是数据本体。
> - 传指针/Map/Slice/Channel：拷贝的是头部的管理结构或者地址数字本身（轻量级拷贝）。

### 30. Go 语言中的引用类型包含哪些？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，我们通常把**“内部包含了指向底层数据结构指针的非标量类型”**统称为引用类型。
> 主要包含：**`Slice`（切片）、`Map`（字典）、`Channel`（通道）**。广义上也可以将**指针**与**接口**归纳进来。

#### 一、 核心引用类型

这三种类型最显著的共同特征是：它们必须使用 `make` 函数来进行创建和深度初始化。它们的变量本身就像一把“钥匙”，真正的数据都存在钥匙背后的底层数据结构里。

1. **切片（Slice）**：
   切片本身是一个占用 24 字节的结构体（`SliceHeader`），里面包含了一个指向底层数组的指针、长度（`len`）和容量（`cap`）。
2. **字典（Map）**：
   Map 的变量本质上是一个指向底层 `hmap` 结构体的指针（在编译器的底层，`map[k]v` 实际上就是 `*hmap`）。
3. **通道（Channel）**：
   与 Map 类似，Channel 变量在底层也是一个指向 `hchan` 结构体的指针。

#### 二、 广义上的引用类型

- **指针（Pointer）**：最为直接的引用类型。它里面存储的就是另一个变量的原始物理内存地址。
- **接口（Interface）**：双指针结构体（如 `iface` 或 `eface`），包含了一个指向类型元数据的指针和一个指向实际数据的指针。当接口传递时，底层的实际数据如果是指针，也会表现出引用的特性。

### 31. Channel 的底层结构是什么？

> [!TIP]
> **核心回答：**
> Go 语言的 `channel` 在底层主要依赖于一个名为 **`hchan`** 的结构体（位于 Go 源码的 `runtime/chan.go` 中）。它在底层其实是一个**带锁的环形队列**，并且维护了发送与接收的等待协程队列。

#### 一、 `hchan` 结构体核心组件

```go
type hchan struct {
    qcount   uint           // 当前队列中剩余的元素个数 (len)
    dataqsiz uint           // 环形队列的槽位总大小，即容量 (cap)
    buf      unsafe.Pointer // 指向底层连续内存数组的指针（仅对有缓冲的 chan 有效）
    elemsize uint16         // 元素的大小
    closed   uint32         // chan 是否关闭的标志 (0:打开, 1:关闭)
    elemtype *_type         // 元素的类型
    sendx    uint           // 缓冲区中下一个写入（发送）的槽位索引
    recvx    uint           // 缓冲区中下一个读取（接收）的槽位索引
    recvq    waitq          // 接收等待队列：因读而阻塞的 Goroutine 双向链表
    sendq    waitq          // 发送等待队列：因写而阻塞的 Goroutine 双向链表
    lock     mutex          // 互斥锁，保护 hchan 的并发安全
}
```

#### 二、 底层三大核心组件

1. **环形队列（`buf` + `sendx` + `recvx`）**：
   对于有缓冲通道，底层会开辟一块连续内存作为循环队列。`buf` 指向这个队列的数组首地址，`sendx` 记录发送游标，`recvx` 记录接收游标。
2. **等待队列（`recvq` 与 `sendq`）**：
   - `recvq`（接收队列）：当通道里没数据，而某个 Goroutine 试图读取（`<-ch`）时，该协程就会被打包成一个 `sudog` 结构体，挂在 `recvq` 链表上。
   - `sendq`（发送队列）：当通道满了（或者无缓冲），某个 Goroutine 试图写入（`ch <-`）时，该协程就会被挂在 `sendq` 链表上。
3. **互斥锁（`lock`）**：
   Go 语言并发安全依然是靠锁（`mutex`）来实现的。向 `chan` 发送或接收数据，进来的第一件事都是加锁 `lock(&c.lock)`，操作完了再解锁。

#### 三、 零拷贝优化机制

当一个消费协程（Goroutine #2）执行 `val := <-ch` 试图读取数据时，如果发现发送等待队列（`sendq`）中有正在阻塞等待的协程（Goroutine #1）：
- **黑魔法**：Go 运行时为了追求极致性能，不会把数据先拷贝到 `buf` 缓冲区再让接收者去拿。
- **直接拷贝**：接收者（Goroutine #2）会直接越过缓冲区，利用指针把发送者（Goroutine #1）`sudog` 节点里的数据直接拷贝到自己的栈内存中。随后唤醒发送者。整个过程减少了一次内存拷贝。

### 32. 使用 range 迭代 map 是有序的吗？

> [!CAUTION]
> **核心回答：**
> **绝对无序，且是底层故意设计成无序的。**
> 即使你的 map 里的数据完全没有变，多次运行 `for range` 遍历，得到的顺序也会不一样。

#### 一、 底层的“防呆”设计：随机种子

为了防止开发者依赖特定顺序去编写业务逻辑，Go 在底层的 `mapiterinit`（迭代器初始化函数）中做了一个故意为之的设计：
- 每次启动 `for range` 遍历时，Go 运行时都会在底层随机生成一个遍历种子。
- 根据这个随机种子，它会随机选择一个桶（Bucket）作为遍历的起点，并且在桶内部也是随机选择一个偏移位置（Offset）开始往后走。

#### 二、 哈希表本身的无序本质

- **哈希散列**：数据的存放位置取决于 `Hash(Key)` 算出来的结果，物理存储本就无序。
- **扩容搬迁**：随着数据量的增加，map 会触发扩容。扩容时旧桶里的元素会被重新洗牌分流（有的留在低位桶，有的搬去高位桶）。同一个 Key 在内存中的绝对位置随时都在变。

#### 三、 工业界如何应对？

如果需要按照特定顺序（如字母顺序 A-Z）读取 Map，标准的解决方案是**“提取 Key 排序法”**：
```go
package main

import (
	"fmt"
	"slices"
)

func main() {
	balances := map[string]float64{
		"ETH": 1.5,
		"BTC": 0.1,
		"SUI": 500.0,
	}

	// 1. 提取所有的 Key 放入切片
	keys := make([]string, 0, len(balances))
	for k := range balances {
		keys = append(keys, k)
	}

	// 2. 对切片进行显式排序 (按字母 A-Z)
	slices.Sort(keys)

	// 3. 按照排好序的 Key 顺序去读取 map
	for _, k := range keys {
		fmt.Printf("Token: %s, Balance: %.2f\n", k, balances[k])
	}
}
```

### 33. slice 的扩容机制是什么？

> [!TIP]
> **核心回答：**
> 当调用 `append` 往切片塞入新元素，且当前容量（`cap`）已经满了时，Go 就会在底层触发扩容。
> - **Go 1.18 以前**：容量小于 1024 时翻倍，大于 1024 时每次增加 1.25 倍。
> - **Go 1.18 及以后**：临界点降为 256。小于 256 直接翻倍；大于等于 256 则采用渐进式平滑扩容公式计算预估容量。
> - 计算出预估容量后，Go 还会进行**内存对齐微调**，向上匹配最接近的固定规格槽位，以此确定最终真正的容量。

#### 一、 核心扩容算法（Go 1.18+）

当一个切片由于 `append` 导致空间不够，需要计算期望的新容量（`newcap`）时，Go 底层的 `growslice` 函数遵循以下递进逻辑：

1. 如果 **「期望容量」** 大于 **「旧容量的 2 倍」**：
   - 新容量直接等于「期望容量」。
2. 如果不满足上述条件：
   - **情况 A**：如果旧容量 $< 256$ 👉 直接翻倍扩容（`newcap = oldcap * 2`）。
   - **情况 B**：如果旧容量 $\ge 256$ 👉 开启渐进式平滑扩容。每次按照以下公式增加：
     $$newcap = oldcap + \frac{oldcap + 3 \times 256}{4}$$
     直到新容量大于或等于期望容量为止。

> **💡 为什么要改掉老版本的“1024 临界点”？**
> 在旧版本中，1024 附近扩容系数会发生剧烈断崖式下跌（从 2.0 倍骤降到 1.25 倍）。新算法由于分子加上了 $3 \times 256 = 768$，使得整个扩容系数曲线随着容量变大平滑地向 1.25 递减，消除了性能突变。

#### 二、 内存对齐微调

编译器拿到「预估新容量」，乘以单个元素占用的字节数，算出预估需要的总内存字节数。Go 运行时在底层内存分配（`mcache`）中，为了防止内存碎片，预先规划好了一系列固定的内存规格槽位（Size Classes，如 8, 16, 32, 48, 64 ... 字节）。
Go 会把预估总字节数，向上匹配一个最接近的固定内存槽位，再反向除以单个元素的大小，得到最终真正的 `cap`。

##### 💻 实例拆解：
假设 `s := []int{1, 2}`（len=2, cap=2），再 `append` 一个元素触发常规扩容：
- **公式预估**：小于 256，直接翻倍。预估 `newcap = 2 * 2 = 4`。
- **计算总字节**：`int` 在 64 位系统下占 8 字节。总字节 = `4 * 8 = 32` 字节。
- **内存匹配**：32 字节刚好卡在 Go 官方的固定规格槽位上，不需要额外向上多要空间。
- **最终容量**：`32 / 8 = 4`。最终 `cap` 就是 4。

#### 三、 扩容时的内存行为

计算好容量并申请到新内存后，切片在底层的行为非常类似于“搬家”：
1. **申请新家**：在堆内存中开辟一块完全独立的连续新数组空间。
2. **搬运家具**：将旧数组里的老元素拷贝（Copy）到新数组的对应槽位中。
3. **切断联系**：让切片内部的 `ptr` 指针指向这个新数组地址。
4. **旧家拆迁**：原来的旧底层数组若无其他切片引用，在下一次 GC 时会被彻底释放。

### 34. Go 语言中的指针运算有哪些？

> [!IMPORTANT]
> **核心回答：**
> 为了保证内存安全，Go 语言在语法层面上**禁止**了普通指针的直接加减运算（如 `p++` 或 `p = p + 1`）。
> 如果必须进行指针运算（如偏移读取内存），需要通过 `unsafe.Pointer` 桥接，利用 `uintptr` 强转成无符号整型数字，进行加减运算后，再转回普通指针。

#### 一、 指针运算的“三位一体”核心

在 Go 中实现指针运算，必须在以下三种类型之间进行无缝转换：
- **普通指针（`*T`）**：它是类型安全的，不能进行任何数学加减运算，也不能在不同类型间强转。
- **`unsafe.Pointer`（万能指针）**：它是所有指针的中转站。任何类型的普通指针都可以转为 `unsafe.Pointer`，反之亦然。但它本身也不能进行加减运算。
- **`uintptr`（数字指针）**：它本质上就是一个无符号整型数字（大小和当前系统的指针完全一致）。它可以进行任何加减乘除运算。

#### 二、 经典的指针运算实战

##### 场景 1：通过结构体基地址，偏移修改私有字段
```go
package main

import (
	"fmt"
	"unsafe"
)

type Token struct {
	Symbol   string // 占用 16 字节
	Decimals uint8  // 占用 1 字节
}

func main() {
	t := &Token{Symbol: "ETH", Decimals: 18}

	// 1. 拿到结构体的起始地址（即第一个字段 Symbol 的地址）
	basePtr := unsafe.Pointer(t)

	// 2. 获取 Decimals 字段的偏移量
	decimalsOffset := unsafe.Offsetof(t.Decimals)

	// 3. 【核心运算步骤】：
	// unsafe.Pointer -> uintptr(转成数字) -> 加偏移量(数学运算) -> 转回 unsafe.Pointer
	decimalsPtr := unsafe.Pointer(uintptr(basePtr) + decimalsOffset)

	// 4. 转回具体的普通指针类型，并直接修改内存
	p := (*uint8)(decimalsPtr)
	*p = 9 // 把 18 强行改成 9

	fmt.Printf("修改后的代币信息: %+v\n", t) // 输出: {Symbol:ETH Decimals:9}
}
```

##### 场景 2：通过指针循环遍历数组
```go
package main

import (
	"fmt"
	"unsafe"
)

func main() {
	arr := [3]int64{111, 222, 333}

	// 1. 拿到数组第一个元素的指针
	startPtr := unsafe.Pointer(&arr[0])

	// 2. 计算一个 int64 元素占用的内存大小（8 字节）
	step := unsafe.Sizeof(arr[0])

	for i := 0; i < 3; i++ {
		// 【指针运算】：每次循环，数字指针累加 i * 8 字节
		currentPtr := unsafe.Pointer(uintptr(startPtr) + uintptr(i)*step)

		// 解引用打印值
		val := *(*int64)(currentPtr)
		fmt.Printf("索引 %d 的值: %d\n", i, val)
	}
}
```

#### 三、 ⚠️ 致命大坑：GC（垃圾回收）背刺

> **黄金法则**：指针转换与加减运算必须在同一行流水线式完成，绝对不能将 `uintptr` 变量存入临时变量并跨行使用！

```go
// ❌ 错误示范：危险代码！
bPtr := uintptr(unsafe.Pointer(t)) // 把指针转成了普通的数字
// 如果在此期间触发了 GC...
targetPtr := unsafe.Pointer(bPtr + 8) // ❌ 此时这里大概率会发生 SegFault 崩溃！
```
- **原因分析**：`uintptr` 本质上只是个普通的数字，不是指针！垃圾回收器（GC）在扫描内存时，会完全无视 `uintptr`。如果在中间等待期间发生了 GC，GC 一看对象 `t` 没有人引用了（唯一的引用线变成了数字 `bPtr`），就会将 `t` 占用的内存直接回收或搬迁。后面的读取便会发生 SegFault 崩溃。

##### 💡 正确姿势：
```go
// ✅ 正确：合为一体，中间不留给 GC 任何反应时间
targetPtr := unsafe.Pointer(uintptr(unsafe.Pointer(t)) + 8)
```

### 35. Go 语言中类型的值可以修改吗？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，这需要分为变量的可变性和类型本身的“可变性（Mutability）”底层设计：
> - **只读类型（Immutable）**：如 `string`。其底层的物理内存是只读的，任何修改都伴随着内存重新分配与拷贝。
> - **就地可变类型（Mutable）**：如 `Slice`、`Map`、`Struct`。可以通过索引、Key 或指针直接就地修改其内部的成员字段。

#### 一、 绝对不可改的类型：只读类型（Immutable）

##### 1. 字符串（string）
字符串在底层的 `StringHeader` 结构体中，指针指向的是一块只读内存。你绝对无法直接修改字符串的某一个字符。
```go
s := "ETH"
// s[0] = 'B' // ❌ 编译直接报错：cannot assign to s[0]

// 怎么改？必须通过切片转换，底层会发生【内存拷贝】
bytes := []byte(s)
bytes[0] = 'B'
s2 := string(bytes) // s2 变成了 "BTH"，但 s 依然是 "ETH"
```
##### 2. 各种基础标量（int, float, bool）
这些是值类型，修改它们的值本质上是直接把旧的数字擦除，填入新的数字，或者在栈上给变量换一个新的值。

#### 二、 能够直接就地修改的类型：就地可变类型（Mutable）

##### 1. 切片（Slice）与 数组（Array）
```go
s := []int{1, 2, 3}
s[0] = 888 // ✅ 就地修改成功
```
##### 2. 字典（Map）
```go
m := make(map[string]int)
m["sui"] = 1
m["sui"] = 2 // ✅ 直接覆盖旧值，就地修改
```

#### 三、 结构体（Struct）修改的经典陷阱

##### 1. 接收者类型决定的可变性
结构体内部的值能不能改，取决于你拿到的到底是它本体的地址（指针），还是它的一份值拷贝。
```go
type Wallet struct {
    Balance uint64
}

// 值接收者：传入的是钱包的副本
func (w Wallet) UpdateBalance(amount uint64) {
    w.Balance = amount // ❌ 这里改的只是克隆体，外部真正的钱包余额毫无变化
}

// 指针接收者：传入的是钱包在内存中的绝对地址
func (w *Wallet) UpdateBalanceReal(amount uint64) {
    w.Balance = amount // ✅ 隔空修改成功，外部余额同步改变
}
```

##### 2. Map 里的结构体值无法直接修改！
这是 Go 语言中一个非常出名的语法陷阱：
```go
type Player struct {
    Level int
}

func main() {
    m := map[string]Player{
        "Alice": {Level: 10},
    }
    // m["Alice"].Level = 20 // ❌ 编译直接报错: cannot assign to struct field m["Alice"].Level in map
}
```
- **原因**：Map 内部的数据是无序散列的，且会随着扩容随时在内存中搬家。Go 为了防止你拿到一个随时会失效的临时地址去修改属性，直接在编译期禁止了这种写法。
- **解法（标准推荐）**：把 map 的 Value 类型改成结构体指针 `map[string]*Player`。
  ```go
  m := map[string]*Player{
      "Alice": {Level: 10},
  }
  m["Alice"].Level = 20 // ✅ 成功修改
  ```

### 36. 解析 JSON 数据时，默认将数值当作哪种类型？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，使用标准库 `encoding/json` 的 `json.Unmarshal` 函数将数值（整数或浮点数）解析到空接口（`interface{}` 或 `any`）中时，Go 默认会将其统一当作 **`float64`** 类型处理。

#### 一、 为什么默认是 float64？

这是因为 JSON 规范（ECMA-404 标准）本身对数字的设计非常笼统，并没有细分 `int32`、`uint64`、`float32` 等类型，所有的数字统称为 "Number"（数值）。Go 语言为了确保通用的兼容性，便做出了用精度最高、容纳范围最广的 `float64` 来一刀切接收所有 JSON Number 的设计。

#### 二、 经典类型断言崩溃复现

```go
package main

import (
	"encoding/json"
	"fmt"
)

func main() {
	jsonStr := `{"nonce": 100}`

	var data map[string]interface{}
	_ = json.Unmarshal([]byte(jsonStr), &data)

	// 1. 看看 Go 实际把它当成了什么类型
	fmt.Printf("nonce 的底层真实类型: %T\n", data["nonce"]) // 🎯 输出: float64

	// 2. ❌ 错误做法：直接断言为 int 会直接崩溃！
	// nonce := data["nonce"].(int) // panic: interface conversion: interface {} is float64, not int

	// 3. ✅ 正确做法：必须先断言为 float64，再强制转换为需要的整数类型
	nonceFloat := data["nonce"].(float64)
	nonceInt := uint64(nonceFloat)
	fmt.Println("成功拿到安全整数:", nonceInt)
}
```

#### 三、 工业界三大标准解决方案

1. **解法 1：定义精准的结构体（Struct）结构接收（最常用）**
   如果提前知道 JSON 的结构，直接用定义好类型的结构体去接收，Go 的 JSON 编译器会智能地帮你进行精准转换。
   ```go
   type Tx struct {
       Nonce uint64 `json:"nonce"` // 显式声明为 uint64
   }
   var tx Tx
   _ = json.Unmarshal([]byte(jsonStr), &tx) // ✅ 完美直接拿到 uint64 类型的 100
   ```

2. **解法 2：使用 `json.Decoder` 并开启 `UseNumber()` 功能**
   如果你必须使用 `map[string]any` 这种动态容器，可以放弃 `json.Unmarshal`，改用 `json.Decoder` 并调用其 `UseNumber()` 方法。开启后，Go 会把 JSON 中的数字当成一个特殊的 `json.Number` 类型（底层是一个字符串），随后可以安全地转为需要的具体类型，避免精度丢失。
   ```go
   decoder := json.NewDecoder(bytes.NewReader([]byte(jsonStr)))
   decoder.UseNumber() // 💡 开启核心开关
   _ = decoder.Decode(&data)

   numObj := data["block"].(json.Number)
   blockNum, _ := numObj.Int64() // 安全地转为 int64，不损失精度
   ```

3. **解法 3：字符串化传递（高精度数字的终极解法）**
   在处理涉及极高精度的数值（如 Solidity 中的 `uint256` 余额）时，工业界标准的做法是**要求在传输层统一包裹成字符串（String）传递**（例如 `{"balance": "1000000000000000000"}`），到 Go 语言后再使用 `math/big` 包的 `big.Int` 进行解析，以此从根本上杜绝精度灾难。

### 37. Array 类型的值作为函数参数是引用传递还是值传递？

> [!TIP]
> **核心回答：**
> 在 Go 语言中，数组（Array）类型的值作为函数参数时，是**值传递**。
> 当你把一个数组传进函数时，Go 会在内存中**完整地复制一份一模一样的数组副本**交给函数。在函数内部对数组的任何修改，都绝对不会影响到外部原本的数组。

#### 一、 现场代码证明

```go
package main

import "fmt"

// 函数接收一个固定长度为 3 的整型数组
func modifyArray(arr [3]int) {
	arr[0] = 999 // 修改函数内部的副本
	fmt.Println("函数内部的数组:", arr)
}

func main() {
	mainArr := [3]int{10, 20, 30}
	modifyArray(mainArr) // 传递数组（此时会发生全量内存拷贝）
	fmt.Println("函数外部的数组:", mainArr) // 🎯 外部数组毫无变化！
}
```
##### 输出结果：
```plaintext
函数内部的数组: [999 20 30]
函数外部的数组: [10 20 30]
```

#### 二、 底层内存行为：昂贵的拷贝代价

在 Go 语言中，数组是一个值类型，拥有固定长度的连续内存。当你将一个包含大量数据的数组直接作为参数传给函数时，Go 运行时会在内存中强行开辟一块同样大小的新空间，并将旧数组里的每个元素逐一复制过去。频繁的参数传递会导致 CPU 耗费大量时间进行内存拷贝，性能开销极大。

#### 三、 工业界的两个标准平替方案

1. **方案 1：改传数组的指针（`*[N]Type`）**
   把函数参数改成指针类型。因为指针在 64 位系统下永远只占 8 字节，拷贝开销忽略不计，且在函数内修改会直接改变外部原数组。
   ```go
   func modifyArrayWithPtr(arr *[3]int) {
       arr[0] = 999 // 通过指针直接操作外部原内存
   }
   ```

2. **方案 2：使用切片（`[]Type`）—— 绝大多数场景的首选**
   切片本身是引用类型（底层由指向数组的指针、len、cap 组成的 24 字节结构体）。你传切片进函数，Go 同样拷贝了这 24 字节，但因为副本指针依然指向同一个底层数组，你可以极其轻量地实现“就地修改”：
   ```go
   func modifySlice(s []int) {
       s[0] = 999 // 极其轻量（只拷贝了 24 字节），且外部同步生效
   }
   // 调用时直接传入切片
   modifySlice(mainArr[:])
   ```
