创建索引时需要注意什么？

在线上生产环境（特别是高频交易流水、链下索引服务 Indexer 等场景）中，创建索引绝对不是一句简单的 CREATE INDEX 就能了事的。索引是一把标准的双刃剑：它在内存中构建出 O(logN) 的高效查找结构，但也会吞噬磁盘空间、重挫写入性能。

在实际大厂的数据库架构设计中，创建索引需要死死坚守以下 四大核心维度（15条军规），以防引发线上死锁或性能雪崩。

一、 选型红线：哪些列「适合」或「不适合」建索引？

1. 优先考虑高辨识度（高离散度）的列
   核心心法： 索引的过滤性取决于该字段不重复的值的数量。

正面教材： wallet_address、email、order_id。这类字段区分度高，B+ 树能精准分流。

反面教材： gender（性别）、status（状态：0 或 1）。这类区分度极低的字段，建了索引后 B+ 树根本无法有效分流，MySQL 优化器评估后觉得还不如全表扫描，导致索引彻底失效，白白污染磁盘。

2. 严禁给频繁更新的列加索引
   如果某个字段每秒钟都要被修改成百上千次（如短视频的实时 view_count、高频波动的账户 balance），坚决不要建索引。因为每次数据变更，底层都需要实时去拆分、重组、合并那棵 B+ 树，会引发严重的磁盘 I/O 争抢与行锁锁冲突。

3. 控制单表索引的总个数（通常不超过 5 个）
   不要试图给表的每个字段都加索引。索引越多，INSERT / UPDATE / DELETE 的代价就线性暴增。一个表如果超过 5 个以上的辅助索引，就要考虑是不是业务设计或联合索引规划得不够合理。

二、 设计精髓：如何高效设计联合索引？
在复杂的业务场景中，我们通常会把多个字段组合起来建立一个联合索引（Composite Index）。

4. 严格遵守“最左匹配原则”
   假设你创建了联合索引 INDEX(age, name, status)：

WHERE age = 18 AND name = 'Alice' ➡ 有效命中。

WHERE name = 'Alice' AND status = 1 ➡ 直接失效！因为跳过了最左侧的 age，底层的 B+ 树在整体上是无序的，无法走索引查找。

设计技巧： 规划联合索引时，必须把查询频率最高、区分度最高（最安全）的列放在最左边。

5. 尽量追求“覆盖索引（Covering Index）”
   如果一个联合索引包含了你 SELECT 后面需要查询的所有目标字段，MySQL 就能直接在辅助索引树上把数据全部返回，实现 0 回表，速度直接拉满。

6. 联合索引的“雪崩隐患”：范围查询卡死后续
   如果你的联合索引是 INDEX(age, salary, name)，而你的 SQL 写成 WHERE age > 18 AND salary = 5000 AND name = 'Bob'：

注意： 走到 age > 18 这个范围查询之后，后面列的索引将彻底失效！也就是说，salary 和 name 根本无法利用索引进行加速。

破局之道： 在设计联合索引时，要把带有范围查询（如 >, <, BETWEEN, LIKE）的字段，尽量放在联合索引的最右侧。

三、 避坑指南：如何预防“隐式索引失效”？
写 SQL 语句和编写业务逻辑时如果不注意，很容易让原本建好的索引在运行时突然瘫痪：

7. 绝对不在索引列上搞任何数学运算或函数调用
   ❌ 错误： WHERE YEAR(created_at) = 2026（导致索引失效，逼迫数据库走全表扫描）。

正确： WHERE created_at BETWEEN '2026-01-01' AND '2026-12-31'。

8. 字符串类型查询必须加单引号（严防隐式类型转换）
   ❌ 错误： WHERE wallet_address = 123（如果字段是 VARCHAR 却传了数字，MySQL 会隐式调用 CAST 函数逐行强转，导致索引报复性报废）。

正确： WHERE wallet_address = '123'。

9. 模糊查询 LIKE 绝不以百分号开头
   ❌ 错误： WHERE name LIKE '%Alice'（左模糊无法利用 B+ 树的左前缀检索，索引失效）。

正确： WHERE name LIKE 'Alice%'（右模糊，正常走索引）。

四、 🛠️ 生产上线红线：如何在千万级大表上安全创建索引？
在线上拥有千万级、上亿级数据的超级大表上创建索引时，绝对不能随手直接敲常规命令，否则很容易引发线上全表死锁、业务停摆的严重事故。

10. 必须强制使用 Online DDL（无锁建索引）
    在传统的 MySQL 旧版本中，建索引会全表加锁，期间所有的写操作全部阻塞卡死。现代 MySQL（InnoDB 5.6+）支持在线无锁变更，在线上执行大表变更时，业内标准的写法是显式加上控制参数：

SQL
-- 🚀 工业级安全建索引标准写法
ALTER TABLE users ADD INDEX idx_wallet (wallet_address),
ALGORITHM=INPLACE,
LOCK=NONE;
ALGORITHM=INPLACE： 直接在原表的底层 .ibd 文件中就地进行 B+ 树的拆分与构建，不创建临时表，极大节省磁盘 I/O。

LOCK=NONE： 允许线上的高并发业务在建索引期间正常进行读写操作（不加任何锁），真正做到用户“无感知”升级。

11. 尽量避开业务高峰期
    哪怕开启了 LOCK=NONE，在 B+ 树构建和 Row Log 合并的准备阶段和结束阶段，MySQL 依然会短暂申请排他锁（MDL 锁）。如果在业务洪峰期执行，这个短暂的加锁动作可能会被前面的慢查询阻塞，进而引发多线程死锁连锁反应（锁堆积）。因此，大表建索引一律安排在凌晨业务低峰期执行。

12. 考虑磁盘剩余空间是否充足
    虽然 INPLACE 算法不需要复制完整的新表，但是在构建 B+ 树以及记录变更日志（Row Log）时，依然会消耗额外的磁盘空间。如果你的磁盘空间已经亮起红灯（比如剩余空间低于 20%），贸然建索引可能会直接导致磁盘被瞬间写满（Disk Full），进而引发数据库宕机。

💡 终极总结口诀
“离散度低切莫建，频繁更新一边靠；
最左匹配联合用，范围查询放最后；
转换函数索引掉，大表无锁（ONLINE）凌晨搞。”
